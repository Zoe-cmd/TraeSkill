## 1. 产品概述

### 1.1 产品愿景

TaskFlow 是一个面向中小型团队（5-50 人）的企业级任务管理平台。本 PRD 重点描述 V2.0 的核心功能——**智能排期建议**，该功能利用 AI 技术分析团队成员的工作负载、任务优先级和截止日期，自动生成最优的任务排期方案，帮助团队管理者做出更科学的排期决策。

### 1.2 产品定位

| 维度 | 描述 |
|------|------|
| 产品名称 | TaskFlow |
| 功能名称 | 智能排期建议 (Smart Scheduling) |
| 功能版本 | V2.0 |
| 产品类型 | Web 应用 |
| 目标用户 | 中小型团队管理者（Team Lead / Project Manager） |
| 核心价值 | 减少手动排期时间，提升资源利用率，降低延期风险 |

### 1.3 业务背景

在当前 V1.0 版本中，TaskFlow 已经支持手动创建任务、分配任务和设置截止日期。但根据用户反馈和数据分析，存在以下痛点：

| 痛点 | 影响 | 严重程度 |
|------|------|----------|
| 手动排期耗时长 | 管理者平均每周花 3-5 小时手动调整任务排期 | 高 |
| 工作负载不均衡 | 部分成员任务过载，部分成员闲置，资源利用率仅 65% | 高 |
| 延期频繁 | 约 35% 的任务未能在截止日期前完成 | 高 |
| 依赖关系遗漏 | 管理者经常忽略任务间的依赖关系，导致阻塞 | 中 |
| 排期缺乏数据支撑 | 凭经验排期，没有基于历史数据做决策 | 中 |

### 1.4 产品目标

| 目标 | 指标 | 当前值 | 目标值 | 时间框架 |
|------|------|--------|--------|----------|
| 减少排期耗时 | 周均排期耗时 | 3-5 小时 | < 1 小时 | 上线后 1 个月 |
| 提升资源利用率 | 团队成员工作负载均衡度 | 65% | > 85% | 上线后 2 个月 |
| 降低延期率 | 任务按时完成率 | 65% | > 85% | 上线后 3 个月 |
| 提升采纳率 | 排期建议采纳率 | - | > 70% | 上线后 1 个月 |
| 用户满意度 | 智能排期功能 NPS | - | > 40 | 上线后 2 个月 |

## 3. 功能需求

### 3.1 功能描述

智能排期建议功能接收项目上下文数据（任务列表、成员列表、历史数据），通过 AI 模型分析后，输出一组排期建议，用户可查看、调整、保存和应用排期方案。

### 3.2 输入数据

| 输入项 | 数据来源 | 必填 | 说明 |
|--------|----------|------|------|
| 项目 ID | 用户当前所在项目 | 是 | 确定排期范围 |
| 未完成任务列表 | 项目下状态不是 DONE 的任务 | 是 | 包含标题、优先级、预估工时、当前截止日期、当前指派人 |
| 项目成员列表 | 项目成员 | 是 | 包含成员 ID、角色、当前工作负载 |
| 历史完成数据 | 任务完成记录（近 30 天） | 否 | 用于 AI 更准确地估算任务耗时 |
| 排期偏好 | 用户设置 | 否 | 工作日/周末安排、团队每日最大工时、排期截止日期范围 |

### 3.3 处理逻辑

```
┌─────────────────────────────────────────────────────────────┐
│                    智能排期建议处理流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 数据收集                                                │
│     ├── 从数据库获取项目未完成任务                            │
│     ├── 从数据库获取项目成员信息                              │
│     ├── 从 Redis 缓存获取成员当前工作负载                      │
│     └── 从数据库获取近 30 天历史完成数据                       │
│                         │                                   │
│                         ▼                                   │
│  2. 数据预处理                                              │
│     ├── 识别任务依赖关系（分析任务描述中的关联关键词）           │
│     ├── 计算每个任务的优先级权重                               │
│     ├── 估算每个任务的实际工时（含历史数据修正）                 │
│     └── 构建约束条件（成员可用时间、任务依赖、截止日期）         │
│                         │                                   │
│                         ▼                                   │
│  3. AI 模型推理                                             │
│     ├── 构建 Prompt（含任务上下文 + 约束条件）                  │
│     ├── 调用 OpenAI GPT-4o API                              │
│     ├── 解析返回的 JSON 排期方案                              │
│     └── 验证方案合理性（所有任务已分配、无冲突等）               │
│                         │                                   │
│                         ▼                                   │
│  4. 方案后处理                                              │
│     ├── 计算成员工作负载分布                                  │
│     ├── 标注高风险任务（接近截止日期）                          │
│     ├── 生成排期方案摘要（文本描述）                            │
│     └── 存入排期方案表（schedule_plans）                      │
│                         │                                   │
│                         ▼                                   │
│  5. 返回结果                                                │
│     ├── 排期方案 ID                                          │
│     ├── 任务-成员-日期映射表                                  │
│     ├── 成员工作负载数据                                      │
│     ├── 风险提示列表                                         │
│     └── 方案摘要文本                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 输出数据

| 输出项 | 数据类型 | 说明 |
|--------|----------|------|
| 方案 ID | UUID | 唯一标识，用于后续引用和对比 |
| 任务排期映射 | Array | 每个任务的建议开始日期、截止日期、指派人 |
| 成员工作负载 | Array | 每个成员的总工时、分配任务数、负载率 |
| 风险提示 | Array | 高风险任务列表、冲突提示、里程碑预警 |
| 方案摘要 | String | 人类可读的方案描述（如"共分配 25 个任务，预计 2 周完成，张三负载最高达 85%"） |

### 3.5 功能细节

| 功能点 | 描述 | 优先级 | 验收标准 |
|--------|------|--------|----------|
| 一键生成排期 | 在项目页面点击按钮，自动生成排期方案 | P0 | 3 秒内返回结果，方案包含所有未完成任务 |
| 甘特图展示 | 以甘特图形式展示排期方案，支持按人/按日期筛选 | P0 | 甘特图正确渲染，支持缩放和拖拽 |
| 工作负载可视化 | 以柱状图展示每个成员的工作负载分布 | P0 | 图表正确渲染，悬停显示详细信息 |
| 手动调整排期 | 支持拖拽调整日期、更换指派人 | P0 | 调整后实时更新视图，记录修改历史 |
| 方案保存与应用 | 保存方案快照，并可一键应用到项目 | P0 | 应用后所有任务更新，通知相关成员，支持撤销 |
| 冲突检测 | 自动检测时间冲突和资源冲突 | P1 | 冲突项高亮显示，提供 2+ 种解决方案 |
| 里程碑预警 | 检测排期是否可能导致里程碑延期 | P1 | 提前 3 天预警，标记受影响的里程碑 |

| 计划 vs 实际分析 | 对比排期方案与实际执行结果 | P2 | 展示偏差率，生成偏差分析报告 |
| 个人今日待办 | 团队成员查看按优先级排序的今日任务 | P1 | 按优先级排序，显示建议完成时间 |

### 3.6 业务规则

| 规则编号 | 规则描述 | 触发条件 | 预期结果 |
|----------|----------|----------|----------|
| BR-S01 | 单个成员每日分配工时不超过 8 小时 | 排期生成 | 超过 8 小时的任务自动延后分配 |
| BR-S02 | 高优先级(urgent)任务优先排在当天 | 排期生成 | urgent 任务排在最前面 |
| BR-S03 | 有依赖关系的任务按依赖顺序排列 | 排期生成 | 前置任务完成日期必须早于后置任务开始日期 |
| BR-S04 | 排期方案生成后 7 天内未被应用则自动过期 | 方案保存后 7 天 | 过期方案标记为 expired，不可再应用 |
| BR-S05 | 每个项目最多同时保存 5 个未应用方案 | 新方案生成时 | 超过 5 个时自动删除最早未应用的方案 |
| BR-S06 | 用户只能排期自己管理的项目 | 任何排期操作 | 非管理员/管理者角色返回 403 |

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 | 测量方法 | 降级策略 |
|------|--------|----------|----------|
| 排期生成响应时间 | < 3 秒 (P95) | APM 工具监控 | 超过 5 秒返回"正在生成"提示，异步轮询结果 |
| 甘特图渲染时间 | < 1 秒 (50 个任务) | 前端 Performance API | 超过 50 个任务时启用虚拟滚动 |
| AI API 调用超时 | < 10 秒 | 后端 timeout 设置 | 超时后返回降级方案（简单规则排期） |
| 并发排期请求 | 支持 10 个并发 | 压力测试 | 超过并发数时排队处理 |

### 4.2 安全需求

| 需求 | 说明 | 实现方式 |
|------|------|----------|
| 数据隔离 | 用户只能查看和排期自己参与的项目 | 后端 API 校验项目成员身份 |
| AI 数据脱敏 | 发送给 OpenAI 的数据不包含用户真实姓名、邮箱等敏感信息 | 使用用户 ID 替代姓名，数据预处理时脱敏 |
| 操作审计 | 所有排期方案的创建、修改、应用操作需记录日志 | 审计日志表记录操作人、时间、操作类型 |
| API 限流 | 防止恶意调用 AI 排期接口 | 单用户每分钟最多 5 次 AI 排期请求 |

## 5. 优先级定义

| 优先级 | 定义 | 包含功能 |
|--------|------|----------|
| P0 | MVP 必须功能，上线前必须完成 | 一键生成排期、甘特图展示、工作负载可视化、手动调整、方案保存与应用 |
| P1 | 重要功能，上线后 1 个月内完成 | 冲突检测、里程碑预警、个人今日待办、方案历史快照 |
| P2 | 增值功能，上线后 2 个月内完成 | 方案对比、计划 vs 实际分析、高级排期偏好设置 |

### 5.1 MVP 范围

MVP 聚焦于核心排期流程：**输入数据 → AI 分析 → 排期方案 → 人工调整 → 应用方案**。具体包括：

- 在项目详情页提供"智能排期"入口按钮
- 支持一键生成排期方案（调用 OpenAI API）
- 甘特图展示排期方案
- 成员工作负载柱状图
- 支持手动拖拽调整任务日期和指派人
- 支持保存方案并一键应用
- 应用后通知相关成员
