# Task Standard - 统一任务规范

## 概述

本文档定义了所有 AI Agent 在管理任务时必须遵守的统一任务规范。Todo 清单是项目执行的"中枢神经系统"，追踪所有任务的状态。

## 适用范围

- 本规范适用于所有 AI Agent 的任务管理
- 所有 Agent 在创建、更新、完成任务时必须遵守本规范

## 语言规范

本规范文档及所有相关文档、代码注释、提交信息均使用简体中文编写。任务标题和描述使用中文。技术术语（如 API、REST、SQL、JSON、HTTP、Git、CI/CD、PRD 等）保留英文原文。

## 任务管理原则

| 原则 | 说明 |
|------|------|
| 原子性 | 每个任务是一个不可再分的工作单元 |
| 可估算 | 每个任务有预估的工作量 |
| 可验证 | 每个任务有明确的完成标准 |
| 可追踪 | 每个任务的状态可追踪 |
| 有归属 | 每个任务有明确的负责人 |
| 有优先级 | 每个任务有明确的优先级 |

## 任务优先级

| 优先级 | 说明 | 示例 |
|--------|------|------|
| P0 | 紧急，阻塞其他任务 | 安全漏洞修复、生产环境故障 |
| P1 | 重要，当前阶段必须完成 | 核心功能开发、关键文档编写 |
| P2 | 一般，本阶段应该完成 | 辅助功能、优化 |
| P3 | 低，可延后 | 体验优化、文档补充 |

## 任务状态

### 状态定义

| 状态 | 说明 |
|------|------|
| NOT_STARTED | 未开始 |
| IN_PROGRESS | 进行中 |
| BLOCKED | 阻塞（等待依赖或人类确认） |
| IN_REVIEW | 审查中 |
| REWORK | 需要返工 |
| COMPLETED | 已完成 |
| SKIPPED | 跳过（不适用于当前项目） |

### 状态转换

```
NOT_STARTED ──▶ IN_PROGRESS ──▶ IN_REVIEW ──▶ COMPLETED
                    │                │
                    │                ▼
                    │            REWORK ──▶ IN_PROGRESS
                    │
                    ▼
                BLOCKED ──▶ IN_PROGRESS
```

## 任务拆分

### 拆分原则

| 原则 | 说明 |
|------|------|
| 1-4 小时 | 每个任务在 1-4 小时内可完成 |
| 独立可测 | 每个任务可独立测试 |
| 有明确输出 | 每个任务有明确的交付物 |
| 无循环依赖 | 任务之间无循环依赖 |

### 拆分示例：用户模块开发

以下是一个真实的用户模块开发任务拆分树，展示了如何将一个大模块拆分为可管理的子任务：

```
用户模块开发
├── TASK-20260711-001: 用户数据模型设计与迁移
│   ├── 设计 users 表结构（字段、约束、索引）
│   ├── 编写 UP/DOWN 迁移脚本
│   ├── 在开发环境执行迁移并验证
│   └── 编写种子数据脚本（10 个测试用户）
│   预估工时: 2h | 输出: 迁移脚本 + 验证截图
│
├── TASK-20260711-002: 用户注册 API 实现
│   ├── 定义 DTO 和验证规则（class-validator）
│   ├── 实现 UserService.register() 方法
│   ├── 实现 UserController POST /auth/register 端点
│   ├── 实现邮箱唯一性校验
│   ├── 实现密码 bcrypt 加密
│   ├── 集成邮件发送服务（异步消息队列）
│   └── 编写单元测试（覆盖率 >= 85%）
│   预估工时: 4h | 输出: 可工作的注册 API + 测试
│
├── TASK-20260711-003: 用户登录 API 实现
│   ├── 实现 UserService.login() 方法
│   ├── 实现 UserController POST /auth/login 端点
│   ├── 实现 JWT Token 生成（access token + refresh token）
│   ├── 实现登录失败次数限制（5 次/15 分钟）
│   └── 编写单元测试
│   预估工时: 3h | 输出: 可工作的登录 API + 测试
│
├── TASK-20260711-004: 用户信息 CRUD API 实现
│   ├── 实现 GET /users/:id（获取用户信息）
│   ├── 实现 PATCH /users/:id（更新用户信息）
│   ├── 实现 DELETE /users/:id（软删除用户）
│   ├── 实现 GET /users（用户列表，分页 + 搜索 + 过滤）
│   ├── 实现权限检查中间件（普通用户只能操作自己的信息）
│   └── 编写单元测试
│   预估工时: 4h | 输出: 可工作的 CRUD API + 测试
│
└── TASK-20260711-005: 用户模块集成测试
    ├── 编写注册-登录-获取信息完整流程测试
    ├── 编写并发注册测试（验证唯一约束）
    ├── 编写异常场景测试（无效邮箱、弱密码、重复注册）
    └── 编写权限测试（普通用户无法访问管理接口）
    预估工时: 3h | 输出: 集成测试套件
```

### 针对不同规模任务的拆分策略

**小型任务（< 2 小时）** — 不再拆分，直接执行

示例：为 User 模型添加一个 phone 字段
- 编写迁移脚本（15 分钟）
- 更新 User 模型定义（10 分钟）
- 更新 API 响应格式（15 分钟）
- 更新测试（20 分钟）

**中型任务（2-8 小时）** — 拆分到 1-2 小时粒度的子任务

示例：实现用户认证模块
- 拆分为：注册、登录、Token 刷新、密码重置、邮箱验证 5 个子任务

**大型任务（> 8 小时）** — 先拆分为中型任务，再拆分为小型任务

示例：实现完整的用户管理系统
- 第一层：数据模型、认证 API、用户信息 API、权限管理、集成测试
- 第二层：每个子项再拆分到 1-2 小时粒度

## 任务优先级定义和示例

### 优先级详细定义

| 优先级 | 定义 | 必须满足条件 | 典型场景 |
|--------|------|-------------|----------|
| P0 | 紧急且阻塞 | 阻塞其他任务或影响系统可用性 | 生产环境故障、安全漏洞、关键路径阻塞 |
| P1 | 重要且紧急 | 当前迭代必须完成，影响核心功能交付 | 核心功能开发、需求变更、关键文档 |
| P2 | 重要但不紧急 | 当前迭代应该完成，可协商调整 | 辅助功能、性能优化、代码重构 |
| P3 | 低优先级 | 可以延后到后续迭代 | 体验优化、文档补充、技术债务清理 |

### 优先级判断矩阵

当需要判断任务优先级时，使用以下矩阵：

| 阻塞性 \ 紧急性 | 紧急 | 不紧急 |
|-----------------|------|--------|
| 阻塞其他任务 | P0 | P1 |
| 不阻塞其他任务 | P1 | P2/P3 |

### 优先级实际示例

**P0 示例**：修复生产环境数据库连接池耗尽导致服务不可用
- 影响：所有用户无法使用系统
- 触发条件：线上告警，错误率 > 5%
- 处理时限：立即处理，30 分钟内修复

**P1 示例**：实现订单创建 API
- 影响：核心业务流程，没有它用户无法下单
- 依赖关系：支付模块、库存模块依赖此任务
- 处理时限：当前迭代结束前必须完成

**P1 示例**：编写 API 规范文档
- 影响：前后端开发都需要参考此文档
- 依赖关系：前端开发、集成测试依赖此任务
- 处理时限：开发开始前必须完成

**P2 示例**：优化用户列表查询性能（添加索引）
- 影响：当前性能可接受，但数据量增长后可能变慢
- 不阻塞：不阻塞任何功能开发
- 处理时限：当前迭代内完成

**P3 示例**：统一错误消息的措辞风格
- 影响：不影响功能，仅改善用户体验
- 不阻塞：不阻塞任何任务
- 处理时限：可延后到下一迭代

## 任务状态流转详细说明

### 完整状态流转图

```
                    ┌──────────────────────────────────────┐
                    │                                      │
                    ▼                                      │
              NOT_STARTED ──────▶ IN_PROGRESS ──────▶ IN_REVIEW
                    ▲                  │                    │
                    │                  │                    ├──▶ COMPLETED
                    │                  │                    │
                    │                  ▼                    ├──▶ REWORK ──▶ IN_PROGRESS
                    │              BLOCKED                  │
                    │                  │                    │
                    │                  ▼                    │
                    │              IN_PROGRESS              │
                    │                                      │
                    └──────────────────────────────────────┘
                    
                    NOT_STARTED ──────▶ SKIPPED
```

### 状态转换规则

| 从 | 到 | 条件 | 触发者 |
|------|------|------|--------|
| NOT_STARTED | IN_PROGRESS | 任务开始执行 | 负责人 |
| NOT_STARTED | SKIPPED | 任务不适用于当前项目 | Tech Lead |
| IN_PROGRESS | IN_REVIEW | 任务完成，提交审查 | 负责人 |
| IN_PROGRESS | BLOCKED | 依赖的任务未完成或等待外部确认 | 负责人 |
| BLOCKED | IN_PROGRESS | 阻塞原因解除 | 负责人 |
| IN_REVIEW | COMPLETED | 审查通过 | Reviewer |
| IN_REVIEW | REWORK | 审查不通过，需要修改 | Reviewer |
| REWORK | IN_PROGRESS | 开始修改 | 负责人 |
| COMPLETED | — | 终态，不可再变更 | — |
| SKIPPED | — | 终态，不可再变更 | — |

### 状态转换中的应用场景

**场景 1：正常流程**

TASK-001 从 NOT_STARTED 开始，负责人拉取任务后变为 IN_PROGRESS，完成开发后提交审查变为 IN_REVIEW，审查通过后变为 COMPLETED。

**场景 2：需要返工**

TASK-002 提交审查后，Reviewer 发现 3 个 Medium 问题，将状态改为 REWORK。负责人修复问题后，状态先回到 IN_PROGRESS，修复完成后再次提交审查变为 IN_REVIEW，最终审查通过变为 COMPLETED。

**场景 3：因依赖阻塞**

TASK-003 开发到一半时，发现依赖的 TASK-001 尚未完成，状态变为 BLOCKED。TASK-001 完成后，状态恢复为 IN_PROGRESS，继续开发。

**场景 4：任务被跳过**

TASK-004 原本计划实现社交登录功能，但产品经理决定 MVP 阶段不包含此功能，状态从 NOT_STARTED 直接变为 SKIPPED。

## 任务估算和跟踪方法

### 估算方法

**方法 1：三点估算法**

使用最乐观、最可能、最悲观三个值估算工作量：

```
预估工时 = (乐观估计 + 4 × 最可能估计 + 悲观估计) / 6

示例：用户注册 API 开发
- 乐观估计: 2h（一切顺利，无阻塞）
- 最可能估计: 3h（正常开发，有小问题）
- 悲观估计: 6h（遇到技术难点或第三方依赖问题）
- 预估工时 = (2 + 4×3 + 6) / 6 = 3.3h ≈ 3.5h
```

**方法 2：类比估算法**

参考类似任务的历史数据估算：

```
示例：用户登录 API 预估
- 参考：用户注册 API 实际耗时 3h
- 调整：登录比注册简单（无需邮件验证）
- 预估：2h
```

**方法 3：自下而上估算法**

将任务拆分为子任务，分别估算后汇总：

```
示例：用户模块开发
- 子任务 1: 数据模型设计 — 2h
- 子任务 2: 注册 API — 3h
- 子任务 3: 登录 API — 2h
- 子任务 4: CRUD API — 3h
- 子任务 5: 集成测试 — 2h
- 总计: 12h + 20% 缓冲 = 14.4h ≈ 15h
```

## 任务依赖

### 依赖类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 完成-开始 (FS) | 前序任务完成后才能开始 | 架构设计完成后才能开始开发 |
| 开始-开始 (SS) | 前序任务开始后才能开始 | 后端开始开发后前端才能开始 |
| 完成-完成 (FF) | 前序任务完成后才能完成 | 测试完成后才能部署 |

### 依赖管理

- 在任务字段中明确标注依赖
- 依赖任务阻塞时，当前任务状态设为 BLOCKED
- 依赖任务完成后，解除阻塞

## 任务统计

### 统计指标

| 指标 | 说明 |
|------|------|
| 总任务数 | 所有任务的总数 |
| 完成任务数 | 状态为 COMPLETED 的任务数 |
| 进行中任务数 | 状态为 IN_PROGRESS 的任务数 |
| 阻塞任务数 | 状态为 BLOCKED 的任务数 |
| 完成率 | 完成任务数 / 总任务数 |
| 进度偏差 | 实际完成日期 vs 计划完成日期 |

## 反模式

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 巨型任务 | 任务太大，无法估算 | 拆分为小任务 |
| 无明确输出 | 任务没有明确交付物 | 定义明确的输出 |
| 无依赖管理 | 忽略任务依赖 | 明确标注依赖关系 |
| 状态不同步 | 任务状态与实际不符 | 及时更新任务状态 |
| 无优先级 | 所有任务同一优先级 | 明确优先级排序 |
