# Handoff Standard - 统一交接规范

## 概述

本文档定义了所有 AI Agent 在交接工作给下一个 Agent 时必须遵守的统一交接规范。交接是 Multi-Agent Collaboration 的核心环节，确保信息完整传递，工作无缝衔接。

## 适用范围

- 本规范适用于所有 AI Agent 之间的工作交接
- 所有 Agent 在完成工作后必须执行交接

## 语言规范

本规范文档及所有相关文档、代码注释、提交信息均使用简体中文编写。交接文档内容使用中文。技术术语（如 API、REST、SQL、JSON、HTTP、Git、CI/CD、PRD、ADR 等）保留英文原文。

## 交接原则

| 原则 | 说明 |
|------|------|
| 完整性 | 交接包含所有必要信息，接收方无需额外询问 |
| 结构化 | 交接使用标准格式，便于接收方解析 |
| 及时性 | 工作完成后立即交接，不积压 |
| 可追溯 | 交接记录可追溯，知道谁交接了什么 |
| 责任明确 | 交接双方的责任明确 |

## 交接流程

### 标准流程

```
1. 当前 Agent 完成工作
   ↓
2. 当前 Agent 通过 Quality Gates
   ↓
3. 当前 Agent 编写交接文档
   ↓
4. 当前 Agent 更新 Todo 状态
   ↓
5. 当前 Agent 更新 Decision Log
   ↓
6. 当前 Agent 发出交接通知
   ↓
7. 下一个 Agent 接收交接
   ↓
8. 下一个 Agent 确认交接内容
   ↓
9. 下一个 Agent 开始工作
```

### 交接时序

    │                       │                      │
    │  完成工作              │                      │

## 交接内容

### 必须包含

| 内容 | 说明 |
|------|------|
| 交付物清单 | 所有输出文档和代码的路径和状态 |
| 关键决策 | 影响后续阶段的决策 |
| 已知问题 | 已知但未解决的问题及原因 |
| 风险提示 | 可能影响后续阶段的风险 |
| 假设说明 | 当前阶段所做的假设 |

### 可选包含

| 内容 | 说明 |
|------|------|
| 下一步建议 | 对后续阶段的建议 |
| 技术细节 | 接收方需要了解的技术细节 |
| 注意事项 | 接收方需要特别注意的事项 |

## 交接确认

### 接收方职责

1. 读取所有交付物
2. 理解关键决策和假设
3. 评估已知问题和风险
4. 确认或提出澄清请求
5. 开始工作

### 澄清流程

如果接收方对交接内容有疑问：

```
1. 接收方在交接文档中标注疑问
2. 接收方更新交接状态为 Needs Clarification
3. 交接方收到通知
4. 交接方回答疑问
5. 接收方确认后更新状态为 Confirmed
```

## 不同角色间的交接差异

### 角色交接矩阵

| 交接方 | 接收方 | 交接重点 | 示例场景 |
|--------|--------|----------|----------|
| Product Manager | Solution Architect | 业务需求理解、用户故事、验收标准 | 需求评审完成后，架构师开始设计 |
| Product Manager | Backend Engineer | 功能需求、业务规则、数据需求 | 需求确认后，后端开始开发 |
| Solution Architect | Backend Engineer | 架构决策、技术选型、接口规范 | 架构设计完成后，后端开始开发 |
| Solution Architect | Frontend Engineer | 组件设计、数据流、状态管理策略 | 架构设计完成后，前端开始开发 |
| Backend Engineer | QA Engineer | API 端点列表、测试数据、已知问题 | 开发完成后，QA 开始测试 |
| Backend Engineer | DevOps Engineer | 部署配置、环境变量、依赖服务 | 开发完成后，DevOps 开始部署 |
| QA Engineer | DevOps Engineer | 测试结果、已知问题、风险评估 | 测试完成后，DevOps 决定是否发布 |
| Any Role | Tech Lead | 所有交付物、关键决策、风险 | 任何阶段完成后，Tech Lead 审查 |
| QA Engineer | Bug 修复负责人 | Bug 详情、重现步骤、修复建议、回归验证方法 | QA 发现 Critical/High Bug 时，触发缺陷修复回路 |

### 交接差异示例

**PM → Architect 交接（需求到设计）**

重点关注：业务目标、用户场景、非功能需求、约束条件。交付物以文档为主，强调业务逻辑的完整性和一致性。

**Architect → Engineer 交接（设计到开发）**

重点关注：技术决策理由、接口定义、模块边界、性能要求。交付物包含架构图和 API 规范，强调可执行性和技术细节。

**Engineer → QA 交接（开发到测试）**

重点关注：功能变更列表、API 端点、测试数据、已知问题和限制。交付物包含代码和部署环境信息，强调可测试性。

## 交接失败的处理流程

### 失败场景和处理

| 场景 | 表现 | 处理流程 |
|------|------|----------|
| 交付物缺失 | 交接文档中列出的交付物不存在或不完整 | 1. 接收方标注缺失项目 2. 交接状态改为 Needs Clarification 3. 交接方补充缺失交付物 4. 重新确认 |
| 信息不一致 | 不同交付物中的信息相互矛盾 | 1. 接收方标注矛盾点 2. 交接方澄清并统一信息 3. 更新所有相关文档 4. 重新确认 |
| 质量不达标 | 交付物未通过 Quality Gates | 1. 拒绝接收 2. 退回给交接方 3. 交接方修复后重新提交 4. 重新审查 |
| 接收方无法理解 | 接收方无法理解交接内容 | 1. 接收方列出具体问题 2. 安排补充说明会议 3. 交接方补充文档 4. 重新确认 |
| 交接中断 | 交接方在交接过程中不可用 | 1. Tech Lead 介入 2. 从已有文档中提取信息 3. 记录信息缺口 4. 标记为风险项 |

### 交接失败升级流程

```
交接失败发生
    ↓
交接双方尝试解决（1 小时内）
    ↓
未解决 → 升级到 Tech Lead
    ↓
Tech Lead 评估并协调（2 小时内）
    ↓
未解决 → 升级到 Project Manager
    ↓
Project Manager 决策（4 小时内）
    ↓
记录到决策日志和经验教训
```

## 共享文档锁协议

### 适用场景

当 Phase 3 中前端工程师与 AI 工程师并行执行时，两者可能需要读取或修改同一份共享文档（如 `docs/API规范文档.md`、`docs/架构设计文档.md` 等）。为避免并行修改导致的内容冲突与覆盖，引入共享文档锁协议进行协调。

| 适用条件 | 说明 |
|----------|------|
| 并行阶段 | Phase 3（前端工程师 + AI 工程师并行） |
| 共享文档 | 多个并行角色同时读写的文档，如 `docs/API规范文档.md` |
| 触发时机 | 任一并行角色需要修改（而非只读）共享文档时 |

### 锁机制

修改共享文档前，必须先获取文档锁。锁信息写入 `docs/交接/文档锁-{文档名}.md`，锁文件命名遵循 `shared/documentation-standard.md` 的文件清单管控。

| 步骤 | 操作 |
|------|------|
| 1. 检查 | 修改前检查 `docs/交接/` 下是否存在该文档的锁文件且状态为「锁定中」 |
| 2. 加锁 | 若无有效锁，创建 `docs/交接/文档锁-{文档名}.md`，写入锁信息 |
| 3. 修改 | 获取锁后修改共享文档 |
| 4. 释放 | 修改完成后释放锁（见「释放机制」） |

锁信息包含以下字段：

| 字段 | 说明 |
|------|------|
| 角色 | 发起锁定的 Agent 角色（如 前端工程师、AI 工程师） |
| 锁定时间 | 锁定发起的时间（YYYY-MM-DD HH:MM） |
| 预计释放时间 | 预计完成修改并释放锁的时间（YYYY-MM-DD HH:MM） |

### 检查机制

并行角色在启动时及每次修改共享文档前，必须检查 `docs/交接/` 下是否存在未释放的文档锁。

| 检查时机 | 检查内容 | 处理方式 |
|----------|----------|----------|
| 角色启动时 | 扫描 `docs/交接/` 下所有 `文档锁-*.md` 文件 | 若存在状态为「锁定中」且未超时的锁，标记相关 Todo 为 BLOCKED 或报告用户 |
| 修改文档前 | 检查目标文档是否有有效锁 | 若有有效锁，等待锁释放后再修改；若已超时，见「超时处理」 |

### 释放机制

修改完成后，必须及时释放文档锁，避免阻塞其他并行角色。

| 释放方式 | 操作 | 说明 |
|----------|------|------|
| 删除锁文件 | 直接删除 `docs/交接/文档锁-{文档名}.md` | 推荐，确保锁彻底清除 |
| 标记已释放 | 在锁文件中将「状态」字段更新为「已释放」 | 备选，保留锁记录用于追溯 |

### 超时处理

若锁超过「预计释放时间」仍未释放，视为异常，需人工介入协调。

| 场景 | 处理流程 |
|------|----------|
| 锁超时未释放 | 1. 其他并行角色检测到超时锁 2. 报告用户协调 3. 用户确认后由持有锁的角色释放或强制释放 4. 记录到决策日志 |

### 锁文件格式

锁文件为轻量 Markdown，存放于 `docs/交接/文档锁-{文档名}.md`。示例如下：

```markdown
# 文档锁: {文档名}

| 字段 | 内容 |
|------|------|
| 文档名 | docs/API规范文档.md |
| 锁定角色 | 前端工程师 |
| 锁定时间 | 2026-07-17 10:00 |
| 预计释放时间 | 2026-07-17 12:00 |
| 状态 | 锁定中 |
```

| 字段 | 说明 | 取值 |
|------|------|------|
| 文档名 | 被锁定的共享文档路径 | 如 `docs/API规范文档.md` |
| 锁定角色 | 持有锁的 Agent 角色 | 前端工程师 / AI 工程师 等 |
| 锁定时间 | 锁定发起时间 | YYYY-MM-DD HH:MM |
| 预计释放时间 | 预计释放时间 | YYYY-MM-DD HH:MM |
| 状态 | 锁的当前状态 | 锁定中 / 已释放 |
