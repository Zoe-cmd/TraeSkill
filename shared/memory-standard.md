# Memory Standard - 统一记忆规范

## 概述

本文档定义了所有 AI Agent 在管理长期记忆时必须遵守的统一记忆规范。在 Multi-Agent Collaboration 模式下，记忆管理确保 Agent 之间的知识传递和经验积累。

## 适用范围

- 本规范适用于所有 AI Agent 的记忆管理
- 所有 Agent 在记录和检索记忆时必须遵守本规范

## 语言规范

本规范文档及所有相关文档、代码注释、提交信息均使用简体中文编写。记忆记录内容使用中文。技术术语（如 API、REST、SQL、JSON、HTTP、Git、CI/CD、Markdown 等）保留英文原文。

## 记忆原则

| 原则 | 说明 |
|------|------|
| 持久化 | 记忆必须持久化到 Markdown 文档 |
| 结构化 | 记忆使用结构化格式，便于检索 |
| 可检索 | 记忆有索引和标签，便于查找 |
| 非冗余 | 同一信息不重复记录 |
| 时效性 | 记忆有创建时间，过期信息可标记 |

### 示例 3：常见问题记忆

```markdown
### MEM-20260711-003: PostgreSQL 连接池耗尽问题

| 字段 | 内容 |
|------|------|
| 记忆 ID | MEM-20260711-003 |
| 类型 | 项目记忆 |
| 标签 | PostgreSQL, 连接池, 性能问题, 故障排查 |
| 创建者 | Database Engineer |
| 创建时间 | 2026-07-11 16:00 |
| 相关决策 | DEC-20260711-008 |

问题现象：生产环境在高峰期出现 "too many clients" 错误，服务不可用持续 3 分钟。

根因分析：
1. 默认连接池最大连接数设置为 20，但实际并发请求峰值达到 200+
2. 部分慢查询（超过 5 秒未完成）占用了连接不放，导致连接池耗尽

解决方案：
1. 将连接池最大连接数调整到 50（根据数据库 max_connections = 200 计算，预留 25% 余量）
2. 添加语句超时设置：statement_timeout = 30000ms
3. 添加连接获取超时：connectionTimeoutMillis = 10000
4. 添加连接池监控指标，在 Grafana 面板中展示活跃连接数、等待连接数

预防措施：在 CI 中添加了连接池压力测试，模拟 200 并发连接场景。
```

### 示例 4：架构决策记忆

```markdown
### MEM-20260711-004: 引入消息队列解耦订单处理

| 字段 | 内容 |
|------|------|
| 记忆 ID | MEM-20260711-004 |
| 类型 | 技术记忆 |
| 标签 | 消息队列, RabbitMQ, 架构解耦, 异步处理 |
| 创建者 | Solution Architect |
| 创建时间 | 2026-07-11 17:30 |
| 相关决策 | DEC-20260711-010 |

决策背景：订单创建后需要同步发送邮件通知、短信通知、更新库存、生成发票四个操作，导致订单创建接口响应时间超过 3 秒。

实施内容：
1. 引入 RabbitMQ 消息队列，订单创建后发送 OrderCreated 事件
2. 四个下游操作各自订阅该事件，独立异步处理
3. 订单创建接口响应时间从 3.2 秒降低到 180ms

实施注意事项：
- 需要处理消息重复消费（幂等性设计），通过订单 ID + 事件类型作为去重键
- 需要处理消息丢失（持久化队列 + 发布确认模式）
- 需要处理服务不可用（死信队列 + 重试机制，最多重试 3 次，间隔 5 秒/30 秒/5 分钟）

效果评估：系统吞吐量提升 40%，订单创建 P95 响应时间从 3.2 秒降低到 180ms，用户体验显著改善。
```

## 记忆检索策略和使用场景

### 检索策略

记忆检索应遵循以下策略，确保高效准确地找到所需信息：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 标签检索 | 通过标签快速定位相关记忆 | 查找特定技术领域的经验 |
| 关键词检索 | 在记忆内容中全文搜索关键词 | 查找具体问题的解决方案 |
| 时间范围检索 | 按创建时间过滤记忆 | 查找近期或特定时期的经验 |
| 关联检索 | 通过决策编号反向查找相关记忆 | 理解某个决策的完整背景 |
| 类型过滤 | 按记忆类型（项目/技术/流程）过滤 | 缩小搜索范围 |

### 使用场景

**场景 1：技术选型时查阅历史经验**

当需要选择新技术方案时，通过标签检索相关技术栈的记忆，查看过往的经验教训和已知问题。例如，选择 Node.js ORM 方案时，检索标签为 `ORM, Node.js, 技术选型` 的记忆，参考之前的 Prisma vs TypeORM 对比评估。

**场景 2：故障排查时参考历史案例**

当遇到生产故障时，通过关键词检索相似问题的记忆，快速定位根因和解决方案。例如，遇到数据库连接超时问题时，检索 `连接池, 超时, 数据库` 相关记忆，参考之前的处理经验。

**场景 3：新成员入职时加速学习**

新成员通过按类型检索项目记忆，快速了解项目技术栈、架构决策、常见陷阱和最佳实践，缩短学习曲线。

**场景 4：阶段评审时回溯决策链条**

在项目阶段评审时，通过关联检索，从决策编号出发，还原整个决策链和相关记忆，全面评估决策效果。

## 记忆清理和过期策略

### 过期标记

记忆不会物理删除，而是通过状态标记管理生命周期：

```markdown
### MEM-20260711-001: 原记忆标题

**状态**: Active（当前有效）

---

### MEM-20260115-003: 旧的缓存策略

**状态**: Deprecated（被 MEM-20260711-005 取代）
**过期原因**: Redis 版本从 5.0 升级到 7.0，缓存策略已更新
**过期日期**: 2026-07-11
```

### 清理策略

| 条件 | 处理方式 | 说明 |
|------|----------|------|
| 技术栈已升级 | 标记为 Deprecated，添加过期原因和替代记忆 | 保留历史记录，便于理解演进过程 |
| 流程已优化 | 标记为 Deprecated，链接到新流程记忆 | 保留旧流程记录，对比改进效果 |
| 信息已过时（超过 1 年） | 标记为 Archived | 归档但保留，不再主动检索 |
| 记忆内容重复 | 标记为 Duplicate，链接到主记忆 | 避免检索结果冗余 |
| 解决方案已失效 | 标记为 Invalid，注明失效原因 | 避免误导后续决策 |

### 定期审查

| 审查频率 | 审查内容 | 负责人 |
|----------|----------|--------|
| 每月 | 检查 6 个月以上未更新的记忆是否需要过期标记 | Tech Lead |
| 每季度 | 审查所有 Deprecated 记忆，确认是否需要归档 | Tech Lead |
| 每半年 | 审查所有记忆分类是否合理，标签是否需要更新 | Tech Lead |

## 记忆冲突处理规则

### 冲突场景

当多条记忆对同一问题给出不同结论时，需要按以下规则处理冲突：

**场景 1：同一技术方案的不同评价**

```
MEM-20260301-005: 推荐使用 MongoDB 存储用户数据（理由：灵活 Schema）
MEM-20260615-012: 不推荐使用 MongoDB 存储用户数据（理由：事务需求增加）
```

**处理规则**：
1. 较新的记忆优先（MEM-20260615-012 > MEM-20260301-005）
2. 旧记忆标记为 Deprecated，注明冲突原因
3. 添加一条新记忆记录冲突的解决结论

**场景 2：不同团队对同一流程的不同实践**

```
MEM-20260401-003: 前端团队使用 Feature Branch 工作流
MEM-20260410-007: 移动端团队使用 Trunk Based 工作流
```

**处理规则**：
1. 两条记忆共存，但需在标签中区分团队（如 `前端团队, 工作流` 和 `移动端团队, 工作流`）
2. 添加一条新记忆说明不同团队选择不同工作流的原因
3. 在记忆内容中明确适用条件

**场景 3：经验教训与后续实践矛盾**

```
MEM-20260101-001: 避免使用 ORM，推荐原生 SQL（理由：性能问题）
MEM-20260501-008: 推荐使用 Prisma ORM（理由：开发效率提升，性能已满足需求）
```

**处理规则**：
1. 旧记忆标记为 Deprecated，注明"性能条件已变化"
2. 新记忆需引用旧记忆，说明为什么之前的经验不再适用
3. 记录决策演进的完整链路

### 冲突解决流程

```
发现冲突
    ↓
评估冲突性质（技术演进 vs 团队差异 vs 错误结论）
    ↓
确定优先级（按时间、按团队、按场景）
    ↓
标记旧记忆状态
    ↓
记录冲突解决结论
    ↓
更新相关标签和索引
```

## 记忆操作

### 记录记忆

```markdown
### MEM-{YYYYMMDD}-{NNN}: {记忆标题}

| 字段 | 内容 |
|------|------|
| 记忆 ID | MEM-{YYYYMMDD}-{NNN} |
| 类型 | 项目记忆 / 技术记忆 / 流程记忆 |
| 标签 | {标签 1}, {标签 2} |
| 创建者 | {Agent Role} |
| 创建时间 | {YYYY-MM-DD HH:MM} |
| 相关决策 | {决策编号} |

{记忆内容}
```

### 检索记忆

通过标签和关键词检索记忆：

```markdown
## 记忆检索

### 标签: {标签名}
- {记忆 ID}: {记忆摘要}
- {记忆 ID}: {记忆摘要}
```

### 更新记忆

记忆不可修改，只能追加新记忆或标记旧记忆为过期：

```markdown
### MEM-20260711-001: 原记忆标题

**状态**: Deprecated（被 MEM-20260712-005 取代）
```

## 反模式

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 记忆不记录 | 经验不记录，下次重复犯错 | 及时记录经验教训 |
| 记忆混乱 | 所有记忆混在一起 | 按类型分类存储 |
| 记忆过期 | 不更新过期的记忆 | 标记过期记忆 |
| 依赖短期记忆 | 依赖聊天上下文 | 持久化到 Markdown 文档 |
