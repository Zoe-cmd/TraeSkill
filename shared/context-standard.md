# Context Standard - 统一上下文规范

## 概述

本文档定义了所有 AI Agent 在管理和传递上下文时必须遵守的统一上下文规范。在 AI Agent First 的工作模式下，上下文管理是确保 Agent 之间有效协作的关键。

## 适用范围

- 本规范适用于所有 AI Agent 的上下文管理
- 所有 Agent 在读取和传递上下文时必须遵守本规范

## 语言规范

本规范文档及所有相关文档、代码注释、提交信息均使用简体中文编写。上下文文档内容使用中文。技术术语（如 API、REST、SQL、JSON、HTTP、Git、CI/CD、Markdown 等）保留英文原文。

## 上下文原则

| 原则 | 说明 |
|------|------|
| 最小上下文 | 只传递必要的上下文，避免信息过载 |
| 结构化 | 上下文使用结构化格式，便于解析 |
| 不可变性 | 上下文一旦产生，不可修改（只追加） |
| 可追溯 | 上下文来源可追溯 |
| 版本化 | 上下文有版本标识 |

### 示例 3：任务上下文文档

```markdown
## 任务上下文: 实现用户注册 API

### 任务信息
- 任务 ID: TASK-20260711-007
- 任务标题: 实现用户注册 API（POST /api/v1/auth/register）
- 负责人: Backend Engineer
- 状态: IN_PROGRESS

### 输入
- API 规范文档 (v1.0.0, line 45-78): 定义了注册接口的请求/响应格式
- 数据库 Schema 文档 (v1.0.0, users 表): 定义了用户表结构
- 架构设计文档 (v1.0.0, section 3.2): 定义了用户模块的职责边界

### 约束
- 密码必须使用 bcrypt 加密，cost factor >= 12
- 邮箱必须验证唯一性（唯一约束 + 应用层检查）
- 注册成功后自动发送欢迎邮件（异步处理，通过消息队列）
- 接口响应时间 P95 < 200ms
- 必须使用参数化查询，防止 SQL 注入

### 假设
- 邮件服务（SendGrid）在注册时可用，SLA 99.9%
- 用户邮箱格式验证在前端已完成，后端仅做二次验证
- 密码强度要求已在前端实施，后端仅验证长度（>= 8字符）

### 已知问题
- 邮件发送失败时不影响注册成功（优雅降级，记录失败日志）
- 并发注册同一邮箱的小概率竞态条件（数据库唯一约束兜底）
```

## 多项目上下文切换规则

### 切换原则

当 Agent 需要在多个项目之间切换时，必须遵循以下规则确保上下文完整恢复：

| 规则 | 说明 |
|------|------|
| 完整保存 | 切换前必须保存当前项目的完整上下文状态 |
| 明确标记 | 每个项目的上下文文件必须明确标记项目名称 |
| 索引管理 | 维护一个项目上下文索引，记录每个项目的当前状态 |
| 隔离存储 | 不同项目的上下文文件存放在独立目录中 |

### 切换流程

```
当前项目 A 工作
    ↓
保存项目 A 上下文（更新所有状态字段）
    ↓
标记项目 A 当前进度（处理到哪个任务、哪个决策点）
    ↓
加载项目 B 上下文（读取项目 B 的上下文文件）
    ↓
确认项目 B 当前状态（理解上次离开时的进度）
    ↓
开始项目 B 工作
```

## 上下文继承和传递规则

### 继承规则

下游 Agent 在接收上下文时，需要遵循以下继承规则：

**规则 1：完整继承**

接收方必须完整理解上游传递的所有上下文，包括：
- 项目背景和业务目标
- 已完成阶段的关键决策和理由
- 当前阶段的输入输出约束
- 已知问题和风险

**规则 2：选择性追加**

接收方只能在继承的基础上追加新信息，不得修改已有的上下文内容：
- 可以追加新的决策记录
- 可以追加新的假设说明
- 可以追加新的已知问题
- 不得修改或删除上游传递的上下文

**规则 3：冲突标注**

如果接收方发现上游上下文中的信息与实际不符，必须标注冲突而非直接修改：
```markdown
### 上下文冲突标注

**冲突位置**: 阶段上下文 > 约束 > 第 3 条
**原内容**: 接口响应时间 P95 < 200ms
**冲突说明**: 经实际验证，在数据量达到 10 万级别时，P95 响应时间约为 350ms，需要重新评估此约束
**标注人**: Backend Engineer
**标注日期**: 2026-07-15
**建议**: 调整约束为 P95 < 500ms 或优化查询性能
```

### 传递规则

上下文在 Agent 间传递时，必须遵循以下格式：

```markdown
## 上下文传递记录

### 传递信息
| 字段 | 内容 |
|------|------|
| 传递编号 | CTX-20260711-001 |
| 传递日期 | 2026-07-11 18:00 |
| 传递方 | Backend Engineer |
| 接收方 | QA Engineer |
| 传递阶段 | Phase 2（开发完成） → Phase 3（集成测试） |
| 上下文版本 | 2.1.0 |

### 传递内容摘要
- 已完成 4 个核心模块的 API 开发（用户、订单、产品、支付）
- 数据库迁移已全部执行完毕

```

## 上下文快照和恢复机制

### 快照机制

上下文快照用于在关键节点保存完整的状态，以便在需要时恢复到该状态：

```markdown
## 上下文快照: SNAP-20260711-001

### 快照信息
| 字段 | 内容 |
|------|------|
| 快照编号 | SNAP-20260711-001 |
| 快照时间 | 2026-07-11 18:00:00 |
| 触发原因 | Phase 2 开发完成，进入 Code Review 阶段 |
| 快照范围 | 项目上下文 + 阶段上下文 + 所有任务上下文 |

### 快照内容
- 项目阶段: Phase 2（开发完成）
- 任务完成数: 12/15（80%）
- 决策记录数: 23 条
- 已知问题数: 3 条
- 代码提交: commit a7f3b2c（feature/user-auth 分支）
```

### 恢复流程

```
检测到需要恢复上下文
    ↓
选择目标快照编号
    ↓
读取快照内容
    ↓
对比当前状态与快照状态（识别差异）
    ↓
决定恢复策略（完整恢复 / 部分恢复）
    ↓
恢复上下文文件
    ↓
通知相关 Agent 上下文已恢复
    ↓
验证恢复后的上下文完整性
```

### 快照触发条件

| 触发条件 | 说明 |
|----------|------|
| 阶段完成 | 每个 Phase 完成时自动创建快照 |
| 重大决策 | 做出影响架构的重大决策时创建快照 |
| 上下文切换 | 切换到其他项目前创建快照 |
| 手动触发 | 任何 Agent 可以在需要时手动创建快照 |
| 每日备份 | 每天结束时自动创建快照（保留最近 7 天） |

## 上下文预警与漂移防护

### 漂移风险

单个角色在长对话中连续完成大量子任务后，上下文逼近限制，会出现：
- 丧失角色身份定位，开始做其他角色的工作
- 越界修改其他角色目录的代码
- 遗忘已完成的进度，重复或遗漏子任务

### 预警与防护机制

漂移防护由 `shared/worklog-standard.md` 统一定义，本规范仅声明联动关系：

| 机制 | 规范来源 | 与上下文管理的关系 |
|------|----------|---------------------|
| 工作日志 | worklog-standard.md | 子任务级恢复点，替代粗粒度快照 |
| 角色锚定 | worklog-standard.md | 周期性身份重申，防止身份稀释 |
| 目录隔离 | coding-standard.md | 物理硬边界，防止越界修改 |
| 上下文预警 | worklog-standard.md | 主动停止，防止被动溢出 |

### 与快照机制的关系

- Phase 级快照（本规范）用于跨阶段恢复，粒度粗
- 工作日志（worklog-standard.md）用于单角色子任务级恢复，粒度细
- 二者互补：阶段切换用快照，角色内长任务用工作日志

## 上下文传递

### 传递方式

上下文通过 Markdown 文档传递，而非 Agent 间的直接对话：

```
Agent A → Markdown 文档 → Agent B
```

### 传递规则

| 规则 | 说明 |
|------|------|
| 显式传递 | 上下文必须显式写入文档，不依赖隐式理解 |
| 完整传递 | 传递所有必要信息，不假设接收方知道任何背景 |
| 结构化传递 | 使用标准格式，便于接收方解析 |
| 版本标记 | 标注上下文版本，避免使用过期信息 |

### 上下文链

```
Phase 0 上下文 → Phase 1 上下文 → Phase 2 上下文 → ...
```

每个阶段的上下文包含：
1. 前一阶段的上下文摘要
2. 当前阶段的输入/输出
3. 当前阶段的决策和假设

## 上下文管理

### 上下文文件

```
docs/
├── context/
│   ├── project-context.md      # 项目级上下文
│   ├── phase-0-context.md      # Phase 0 上下文
│   ├── phase-1-context.md      # Phase 1 上下文
│   └── ...
```

### 上下文更新

每个 Agent 在完成工作后必须更新上下文：
1. 更新项目上下文的状态
2. 创建下一阶段的上下文
3. 标注当前阶段的完成状态

## 反模式

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 依赖聊天上下文 | Agent 依赖聊天历史获取信息 | 从 Markdown 文档读取上下文 |
| 上下文丢失 | 关键信息未传递到下一阶段 | 完整写入上下文文档 |
| 上下文过载 | 传递过多无关信息 | 只传递必要信息 |
| 上下文不一致 | 多个地方有冲突的信息 | 维护单一来源 |
| 上下文漂移 | 长对话后角色越界改其他角色代码 | 遵守 worklog-standard 的锚定与目录隔离 |
