# 工作流程 - AI Agent 协作工作流

## 概述

本文档定义了整个 AI Agent 团队的协作工作流，包括阶段划分、角色顺序、交接机制、质量门禁和异常处理流程。所有 Agent 必须严格遵守本文档定义的工作流。

## 单角色单对话原则

**这是最重要的工作流规则。**

| 规则 | 说明 |
|------|------|
| 一个对话 = 一个角色 | 每个 AI 对话只激活一个角色。角色完成工作后，对话即结束 |
| 交接后必须停止 | 角色产出交接文档后，必须停止当前对话，不得自动激活下一个角色 |
| 新对话 = 新角色 | 用户在新对话中说「继续」或「激活 {角色名}」来启动下一个角色 |
| 禁止角色串联 | 禁止在同一个对话中完成角色 A 后自动开始角色 B 的工作 |

### 为什么要单角色单对话

| 原因 | 说明 |
|------|------|
| 避免上下文溢出 | 单个角色的 Skill 文件 + 共享规范 + 输入文档已经很占上下文，串联多个角色必然溢出 |
| 确保角色不越界 | 一个对话只有一个角色，天然防止产品经理去做 UI 设计、架构师去做数据库设计 |
| 方便用户管理 | 用户可以在每个角色完成后审查产出物，确认满意后再启动下一角色 |
| 可回溯 | 每个对话对应一个角色的完整工作，便于定位问题 |

### 正确的对话流程

```
对话 1: 用户「启动项目」
  → 协调者激活项目经理
  → 项目经理完成工作，产出 docs/项目计划.md、docs/任务清单.md
  → 项目经理编写交接文档 docs/交接/交接-项目经理-to-技术负责人.md
  → 协调者输出: "✅ 项目经理已完成。下一角色: 技术负责人。请在新对话中说「继续」"
  → 对话结束

对话 2: 用户「继续」
  → 协调者读取交接文档，确定下一角色是技术负责人
  → 激活技术负责人
  → 技术负责人完成工作，产出技术选型决策
  → 技术负责人编写交接文档
  → 协调者输出: "✅ 技术负责人已完成。下一角色: 产品经理。请在新对话中说「继续」"
  → 对话结束

对话 3: 用户「继续」
  → ...以此类推
```

### 错误的对话流程（禁止）

```
对话 1: 用户「启动项目」
  → 协调者激活项目经理... 完成
  → 协调者自动激活技术负责人... 完成    ← ❌ 禁止！
  → 协调者自动激活产品经理... 完成      ← ❌ 禁止！
  → 协调者自动激活 UI/UX 设计师...      ← ❌ 禁止！上下文已溢出，角色开始越界
```

### 单角色内的防漂移

单角色单对话解决了「角色间」串联溢出，但单个角色在长对话中完成大量子任务时，
仍可能发生「角色内」漂移（身份遗忘、目录越界、进度丢失）。防漂移机制由
`shared/worklog-standard.md` 定义，要点：

- 每个角色在对话内维护 `docs/工作日志/工作日志-{角色代号}.md` 作为子任务级恢复点
- 每完成一个子任务执行角色锚定（重申身份 + 越界自检）
- 严格只修改自身代码目录（见 `shared/coding-standard.md` 目录隔离矩阵）
- 触发上下文预警时主动写日志并建议用户开新对话继续（同一角色的新对话）

### 阶段前置条件

每个阶段有明确的前置条件，不满足则不可进入下一阶段。

| 阶段 | 前置条件 | 阻塞条件 |
|------|----------|----------|
| Phase 0 | 项目立项批准 | 无 |
| Phase 1 | Phase 0 完成，Todo 已创建 | 项目经理未完成项目规划 |
| Phase 2 | 系统架构师交付架构文档 | 架构文档未通过技术负责人评审 |
| Phase 3 | 后端工程师交付 API 文档 | API 文档未通过代码评审工程师评审 |
| Phase 4 | Phase 2 和 Phase 3 代码完成 | 代码未通过编译/构建 |
| Phase 5 | 测试工程师测试通过，安全工程师审计通过 | 存在 Critical/High 安全问题 |
| Phase 6 | 生产环境部署成功 | 部署失败或回滚 |

## 详细工作流

### Phase 0: 项目初始化

#### 0.1 项目经理启动

**输入**: 项目需求简述（来自 Human Developer）

**流程**:
1. 读取 `project-manager.md` 了解自身职责
2. 读取 `shared/task-standard.md` 了解任务规范
3. 读取 `templates/todo-template.md` 了解 Todo 格式
4. 分析项目需求，识别项目范围和风险
5. 创建 Todo 文件，定义所有阶段任务
6. 创建 Decision Log 文件（初始化为空）
7. 生成项目规划文档
8. 提交给 Human Developer 审批

**输出**:
- `docs/项目计划.md` — 项目规划文档
- `docs/任务清单.md` — 任务清单
- `docs/决策日志.md` — 决策日志（初始化）

**交接**: 通知 技术负责人 开始工作

#### 0.2 技术负责人启动

**输入**: 项目规划文档、Todo 清单

**流程**:
1. 读取 `tech-lead.md` 了解自身职责
2. 读取 `shared/decision-log-standard.md` 了解决策日志规范
3. 读取项目经理的输出文档
4. 分析技术需求，确定技术栈方向
5. 评估技术债务基线
6. 创建技术选型决策记录
7. 提交给 Human Developer 审批

**输出**:
- `docs/决策日志.md` — 追加技术选型决策

**交接**: 通知 产品经理 开始工作

---

### Phase 1: 需求与设计

#### 1.1 产品经理启动

**输入**: 项目规划文档、技术选型决策

**流程**:
1. 读取 `product-manager.md` 了解自身职责
2. 读取 `templates/prd-template.md` 了解 PRD 格式
3. 读取 `shared/documentation-standard.md` 了解文档规范
4. 读取项目规划文档和技术选型决策
5. 进行需求分析：用户故事、功能需求、非功能需求
6. 编写 PRD 文档
7. 定义验收标准
8. 定义优先级排序
9. 自检测试
10. 更新 Todo 状态
11. 交接给 UI/UX设计师

**输出**:
- `docs/产品需求文档.md` — 产品需求文档

**质量门禁**:
- PRD 必须包含所有必填章节
- 每个功能需求必须有对应的验收标准
- 优先级排序必须有明确的依据

**交接**: 通知 UI/UX设计师 开始工作

#### 1.2 UI/UX设计师启动

**输入**: PRD 文档

**流程**:
1. 读取 `ui-ux-designer.md` 了解自身职责
2. 读取 PRD 文档
3. 分析用户需求，设计信息架构
4. 设计用户流程（User Flow）
5. 设计线框图（Wireframe）
6. 设计高保真原型（Mockup）
7. 定义设计系统（Design System）
8. 定义交互规范
9. 自检测试
10. 更新 Todo 状态
11. 交接给 系统架构师

**输出**:
- `docs/设计系统.md` — 设计系统文档
- `docs/用户流程图.md` — 用户流程图
- `docs/线框图.md` — 线框图说明
- `docs/高保真原型.md` — 高保真原型说明

**质量门禁**:
- 设计必须覆盖所有 PRD 中的用户场景
- 设计系统必须包含颜色、字体、间距、组件规范
- 交互规范必须定义所有交互状态

**交接**: 通知 系统架构师 开始工作

#### 1.3 系统架构师启动

**输入**: PRD 文档、设计系统文档

**流程**:
1. 读取 `solution-architect.md` 了解自身职责
2. 读取 `templates/architecture-template.md` 了解架构文档格式
3. 读取 `shared/api-standard.md` 了解 API 规范
4. 读取 PRD 文档和设计系统文档
5. 分析需求，设计系统架构
6. 确定技术选型（最终确认）
7. 定义模块划分
8. 定义接口边界
9. 定义数据流
10. 定义非功能需求实现方案
11. 编写架构决策记录（ADR）
12. 自检测试
13. 更新 Todo 状态
14. 交接给 数据库工程师

**输出**:
- `docs/架构设计文档.md` — 架构文档
- `docs/决策日志.md` — 追加架构决策记录

**质量门禁**:
- 架构文档必须包含架构图（文字描述或 Mermaid 图）
- 每个模块必须有明确的职责定义
- 接口边界必须清晰定义
- 非功能需求必须有实现方案

**交接**: 通知 数据库工程师 开始工作

### Phase 2: 数据与后端

#### 2.1 数据库工程师启动

**输入**: PRD 文档、架构文档

**流程**:
1. 读取 `database-engineer.md` 了解自身职责
2. 读取 `shared/database-standard.md` 了解数据库规范
3. 读取 `templates/database-template.md` 了解数据库文档格式
4. 读取 PRD 文档和架构文档
5. 分析数据需求，识别实体和关系
6. 设计 ER 图
7. 定义表结构、索引、约束
8. 定义迁移策略
9. 定义数据访问层接口
10. 编写数据库文档
11. 自检测试
12. 更新 Todo 状态
13. 交接给 后端工程师

**输出**:
- `docs/数据库设计文档.md` — 数据库 Schema 文档
- `docs/数据库迁移计划.md` — 数据库迁移计划

**质量门禁**:
- 所有表必须有主键
- 所有外键必须有索引
- 所有字段必须有注释
- 迁移策略必须可回滚

**交接**: 通知 后端工程师 开始工作

#### 2.2 后端工程师启动

**输入**: PRD 文档、架构文档、数据库 Schema 文档、API 规范

**流程**:
1. 读取 `backend-engineer.md` 了解自身职责
2. 读取 `shared/api-standard.md` 了解 API 规范
3. 读取 `shared/coding-standard.md` 了解编码规范
4. 读取 `templates/api-template.md` 了解 API 文档格式
5. 读取所有输入文档
6. 设计 API 接口
7. 编写 API 文档
8. 实现业务逻辑
9. 实现中间件
10. 实现数据访问层
11. 编写单元测试
12. 自检测试
13. 更新 Todo 状态
14. 交接给 前端工程师 和 AI工程师

**输出**:
- `docs/API规范文档.md` — API 规范文档
- 后端代码实现
- 单元测试代码

**质量门禁**:
- 所有 API 必须有文档
- 所有 API 必须有错误处理
- 所有 API 必须有请求验证
- 代码必须通过 Lint 检查
- 单元测试覆盖率 >= 80%

**交接**: 通知 前端工程师 和 AI工程师 开始工作

## 并行执行策略

### 可并行执行的阶段

```
Phase 0:    Project Manager ──── Tech Lead
                │                    │
                └────────┬───────────┘
                         │
Phase 1:    Product Manager ── UI/UX Designer ── Solution Architect
                (顺序执行，不可并行)

Phase 2:    Database Engineer ── Backend Engineer
                (顺序执行，不可并行)

Phase 3:    Frontend Engineer ──── AI Engineer
                (可并行执行)

Phase 4:    QA Engineer ── Security Engineer ── Code Reviewer
                (顺序执行，不可并行)

Phase 5:    DevOps Engineer ── Project Manager
                (顺序执行，不可并行)
```

### 并行执行规则

1. **前端工程师和AI工程师可并行执行** — 两者依赖不同，无直接冲突
   - 前端工程师与 AI 工程师并行时，若需修改共享文档（如 `API规范文档.md`），必须遵守 `shared/handoff-standard.md` 中的「共享文档锁协议」
2. **Phase 0 的项目经理和技术负责人可部分并行** — 技术负责人可以在项目经理完成项目规划后立即开始
3. **所有其他阶段必须顺序执行** — 确保文档链完整

## 异常处理流程

### 阶段回退

如果当前阶段发现问题需要回到前一个阶段：

```
当前阶段发现问题
    │
    ▼
判断问题类型
    │
    ├── 需求问题 → 回到 Product Manager
    ├── 设计问题 → 回到 UI/UX Designer 或 Solution Architect
    ├── 数据问题 → 回到 Database Engineer
    ├── 实现问题 → 回到对应 Backend/Frontend/AI Engineer
    ├── 质量问题 → 回到对应 Engineer 修复
    ├── 安全问题 → 回到对应 Engineer 修复
    └── 部署问题 → 回到 DevOps Engineer 修复
```

### 回退流程

1. 当前 Agent 识别问题并记录在 Decision Log
2. 更新 Todo 状态，标记需要回退的任务
3. 通知前序 Agent 重新启动
4. 前序 Agent 读取问题描述
5. 前序 Agent 修复问题
6. 前序 Agent 更新交付物
7. 前序 Agent 重新交接

### 紧急回退

当发现严重问题（Critical Bug、安全漏洞）时：

1. 立即通知 技术负责人 和 Human Developer
2. 技术负责人评估影响范围
3. Human Developer 决定是否回退
4. 执行紧急回退流程
5. 更新 Decision Log

## 交接规范

### 交接后必须停止

**角色完成交接文档后，必须停止当前对话。** 不得在当前对话中自动激活下一个角色。

```
角色完成工作
    ↓
编写交接文档 → docs/交接/交接-{源角色}-to-{目标角色}.md
    ↓
更新 Todo 状态
    ↓
输出交接摘要（告知用户下一角色名称）
    ↓
停止。等待用户在新对话中触发下一角色
```

### 交接内容

每个 Agent 在交接时必须包含：

| 交接项 | 说明 | 必填 |
|--------|------|------|
| 交接文档 | 当前阶段的完成情况和关键决策 | 是 |
| 交付物 | 所有输出文档和代码 | 是 |
| 已知问题 | 已知但未解决的问题及原因 | 是 |
| 风险提示 | 可能影响后续阶段的风险 | 是 |
| 假设说明 | 当前阶段所做的假设 | 是 |
| 下一步建议 | 对后续阶段的建议 | 否 |

### 交接格式

参见 `shared/handoff-standard.md`

## 工作流状态

### 状态定义

| 状态 | 说明 |
|------|------|
| `NOT_STARTED` | 未开始 |
| `IN_PROGRESS` | 进行中 |
| `BLOCKED` | 阻塞（等待依赖或人类确认） |

```
                    │
                    ▼
                BLOCKED ──▶ IN_PROGRESS
```
