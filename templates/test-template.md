### 缺陷报告格式

```markdown
# Bug Report: BUG-20260711-001

## 缺陷信息

| 字段 | 内容 |
|------|------|
| 缺陷 ID | BUG-20260711-001 |
| 标题 | 用户注册接口在并发请求时可能创建重复用户 |
| 严重程度 | High |
| 优先级 | P0 |
| 状态 | Open |
| 发现版本 | v1.0.0 |
| 发现环境 | 测试环境 |
| 发现日期 | 2026-07-11 |
| 发现人 | QA Engineer |
| 负责人 | Backend Engineer |
| 关联测试用例 | TC-006（并发注册测试） |

## 描述

在高并发场景下，两个相同的邮箱注册请求同时到达服务端，都通过了邮箱唯一性检查，导致创建了两个相同邮箱的用户。

## 重现步骤

1. 准备两个并发请求，使用相同的邮箱和密码
2. 同时发送两个 POST /auth/register 请求
3. 检查数据库中的用户记录

## 重现频率

- 低并发（< 10 QPS）：不重现
- 中并发（10-50 QPS）：约 5% 概率重现
- 高并发（> 50 QPS）：约 20% 概率重现

## 预期结果

只有一个请求成功（201），另一个请求失败（409）。

## 实际结果

两个请求都返回 201，数据库中出现了两条相同邮箱的用户记录。

## 根因分析

应用层的邮箱唯一性检查（SELECT 查询）和 INSERT 操作之间存在竞态条件。虽然数据库有唯一约束，但使用了 `WHERE deleted_at IS NULL` 的部分唯一索引，两个并发请求在检查时都认为邮箱不存在。

## 修复建议

```typescript
// 方案 1：使用数据库行级锁
async register(dto: RegisterDto) {
  return this.dataSource.transaction(async (manager) => {
    // SELECT ... FOR UPDATE 锁定邮箱行
    const existing = await manager.findOne(User, {
      where: { email: dto.email },
      lock: { mode: 'pessimistic_write' }
    });
    if (existing) throw new ConflictException('邮箱已被注册');
    return manager.save(User, userData);
  });
}

// 方案 2：使用 Redis 分布式锁
async register(dto: RegisterDto) {
  const lockKey = `register:${dto.email}`;
  const locked = await redis.set(lockKey, '1', 'NX', 'EX', 10);
  if (!locked) throw new ConflictException('请稍后重试');
  try {
    // 正常注册流程
  } finally {
    await redis.del(lockKey);
  }
}
```

### FAIL-001: 并发注册可能创建重复用户

| 字段 | 内容 |
|------|------|

| 严重程度 | High |
| 状态 | Open |
| 负责人 | Backend Engineer |

**描述**: 高并发场景下，两个相同的邮箱注册请求同时到达，都通过了唯一性检查，导致创建了重复用户。

**重现步骤**:

**实际结果**: 两个都成功（201）

### FAIL-002: 密码重置 Token 在 Redis 重启后失效

| 字段 | 内容 |
|------|------|
| 测试 ID | TC-045 |
| 严重程度 | Medium |
| 状态 | Open |
| 负责人 | Backend Engineer |

**描述**: 密码重置 Token 存储在 Redis 中，Redis 重启后 Token 丢失，用户无法重置密码。

### FAIL-003: 用户列表接口未限制每页最大数量

| 字段 | 内容 |
|------|------|
| 测试 ID | TC-078 |
| 严重程度 | Medium |
