# Decision Log Template - 决策日志模板

## 使用说明

本模板用于所有 Agent 记录关键决策。决策日志是项目的"决策记忆"，记录所有关键技术决策及其理由、替代方案和影响。

### 何时使用

当以下情况发生时，必须记录决策日志：

| 触发条件 | 示例 |
|----------|------|
| 技术选型决策 | 选择数据库、框架、中间件等技术组件 |
| 架构设计决策 | 选择架构风格、模块划分、通信协议 |
| 流程决策 | 选择开发流程、部署策略、审查流程 |
| 设计模式决策 | 选择特定设计模式解决具体问题 |
| 库/工具选择 | 选择第三方库或开发工具 |
| 撤销或变更 | 撤销或变更之前的决策 |

### 决策编号格式

```
DEC-{YYYYMMDD}-{NNN}

YYYYMMDD: 决策日期
NNN: 当天序号，从 001 开始递增
```

### 决策类型分类

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| 架构决策 | 影响系统整体结构的决策 | 选择微服务还是单体架构 |
| 技术选型 | 选择具体技术组件 | 选择 PostgreSQL 还是 MySQL |
| 设计决策 | 影响代码设计和模式的决策 | 选择 Repository 模式还是 Active Record |
| 流程决策 | 影响开发流程的决策 | 选择 Git Flow 还是 Trunk Based |
| 安全决策 | 影响安全策略的决策 | 选择 JWT 还是 Session 认证 |
| 运维决策 | 影响部署和运维的决策 | 选择 Kubernetes 还是 Docker Swarm |

### 决策状态流转

```
Proposed ──────▶ Accepted ──────▶ Implemented
   │                                  │
   │                                  │
   ▼                                  ▼
Rejected                         Deprecated ──▶ Superseded
```

| 状态 | 说明 | 可流转到 |
|------|------|----------|
| Proposed | 提议中，尚未最终决定 | Accepted, Rejected |
| Accepted | 已接受，正在实施 | Implemented |
| Implemented | 已实施完成 | Deprecated |
| Deprecated | 已废弃，不再适用 | Superseded |
| Superseded | 被新决策取代 | —（终态） |
| Rejected | 已拒绝 | —（终态） |

## 决策类型示例

### 示例 1：架构决策

```markdown
## DEC-20260711-001: 选择微服务架构而非单体架构

### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260711-001 |
| 决策日期 | 2026-07-11 |
| 决策者 | Solution Architect |
| 决策状态 | Accepted |
| 决策类型 | 架构决策 |
| 影响范围 | 整个系统架构，影响所有模块 |

### 背景

ShopFlow 电商平台预计在 6 个月内用户量从 0 增长到 10 万，需要支持 4 个团队并行开发（用户团队、订单团队、产品团队、支付团队）。当前阶段需要确定系统的整体架构风格，以指导后续的模块划分和开发计划。

### 问题

选择单体架构还是微服务架构？需要权衡开发效率、运维复杂度、团队协作和扩展性。

### 决策

采用微服务架构，按业务领域拆分为以下 4 个核心服务：
- 用户服务（auth-service）：负责用户注册、登录、认证、信息管理
- 订单服务（order-service）：负责订单创建、查询、状态管理
- 产品服务（product-service）：负责产品管理、库存管理
- 支付服务（payment-service）：负责支付处理、退款管理

### 理由

1. 支持多团队并行开发，每个团队可以独立负责一个服务
2. 支持独立部署和扩展，可根据各服务的流量特点独立扩缩容
3. 技术栈灵活选择，不同服务可以根据需求选择不同的技术方案
4. 故障隔离，单个服务的故障不会影响其他服务
5. 符合团队当前的组织结构（4 个独立团队）

### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| 单体架构 | 开发简单、部署简单、调试方便 | 团队耦合、扩展性差、技术栈锁定 | 无法支持 4 个团队并行开发 |
| 模块化单体 | 代码模块化、部署简单、过渡方案 | 仍有单点故障、扩展性受限 | 团队仍然耦合，无法独立部署 |
| 微服务（已选择） | 独立部署、独立扩展、团队解耦 | 运维复杂度高、分布式事务难处理 | — |

### 影响

正面影响：
- 4 个团队可以独立开发、测试、部署各自的微服务
- 系统可以根据各服务的负载独立扩缩容
- 技术栈可以根据各服务的特点灵活选择

负面影响：
- 需要额外的基础设施（API 网关、服务发现、消息队列）
- 需要制定服务间通信规范和分布式事务处理方案
- 运维复杂度增加，需要统一的监控和日志系统
- 开发环境搭建复杂度增加

受影响方：
- 所有开发团队：需要学习微服务开发规范和工具链
- DevOps 团队：需要搭建和管理微服务基础设施
- QA 团队：需要调整测试策略，增加集成测试和契约测试

### 实施计划

| 步骤 | 描述 | 负责人 | 预计时间 |
|------|------|--------|----------|
| 1 | 搭建 API 网关（Kong）和服务发现（Consul） | DevOps Engineer | 2 天 |
| 2 | 定义服务间通信规范（REST + RabbitMQ） | Solution Architect | 1 天 |
| 3 | 创建 4 个服务的代码仓库和脚手架 | Backend Engineer | 1 天 |
| 4 | 编写微服务开发指南文档 | Tech Lead | 1 天 |
| 5 | 团队培训（微服务开发规范） | Tech Lead | 0.5 天 |

### 后果

决策实施后填写。描述实际后果，包括正面和负面。

### 相关决策

- DEC-20260711-002: 选择 PostgreSQL 作为主数据库 — 每个微服务的数据存储决策
- DEC-20260711-003: 选择 gRPC 作为服务间同步通信协议 — 服务间通信方式
- DEC-20260711-004: 引入 RabbitMQ 实现异步消息 — 服务间异步通信方式

### 审批

| 角色 | 姓名 | 审批状态 | 日期 | 备注 |
|------|------|----------|------|------|
| Tech Lead | 张三 | Approved | 2026-07-11 | 与团队讨论后一致同意 |
| Human Developer | 李四 | Approved | 2026-07-11 | 关注运维复杂度，建议逐步实施 |
```

### 示例 2：技术选型决策

```markdown
## DEC-20260711-002: 选择 PostgreSQL 16 作为主数据库

### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260711-002 |
| 决策日期 | 2026-07-11 |
| 决策者 | Tech Lead |
| 决策状态 | Accepted |
| 决策类型 | 技术选型 |
| 影响范围 | 数据层，影响所有微服务的数据存储 |

### 背景

ShopFlow 平台需要存储结构化数据，包括用户信息、订单数据、产品信息、支付记录等。业务需求包括：复杂查询（多表关联、聚合统计）、事务支持（订单创建涉及多步操作）、JSON 数据存储（产品属性灵活扩展）、全文搜索（商品搜索）。

### 问题

选择哪个关系型数据库作为主数据库？候选方案包括 PostgreSQL、MySQL 和 MariaDB。

### 决策

选择 PostgreSQL 16 作为主数据库，所有微服务统一使用。

### 理由

1. 开源且成熟，社区活跃，文档丰富
2. 支持 JSONB 类型，可以存储灵活的 JSON 数据，同时支持索引和查询
3. 内置全文搜索功能（tsvector/tsquery），无需额外引入 Elasticsearch（MVP 阶段）
4. 支持窗口函数、CTE、LATERAL JOIN 等高级 SQL 特性，满足复杂查询需求
5. 扩展生态丰富（PostGIS 地理信息、TimescaleDB 时序数据、pgvector 向量搜索）
6. 许可证友好（PostgreSQL License），商业使用无限制
7. 支持多种索引类型（B-tree、Hash、GiST、GIN、BRIN），覆盖各种查询场景

### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| MySQL 8.0 | 简单易用，广泛使用，MySQL 人才多 | JSON 支持弱，高级 SQL 特性少，全文搜索功能弱 | 不符合复杂查询和 JSON 灵活存储的需求 |
| MariaDB 11 | MySQL 兼容，开源社区活跃 | JSON 支持不如 PostgreSQL，高级特性少 | 与 MySQL 类似的问题 |
| PostgreSQL 16（已选择） | 功能丰富，扩展性强，JSONB 支持优秀 | 学习曲线稍陡，运维经验要求稍高 | — |

### 影响

正面影响：
- 内置 JSONB 支持，产品属性等灵活字段无需额外 NoSQL 数据库
- 内置全文搜索，MVP 阶段无需引入 Elasticsearch
- 丰富的索引类型，可以覆盖各种查询性能优化需求
- 统一的数据库选型，降低运维和学习的复杂度

负面影响：
- 团队需要学习 PostgreSQL 特有的语法和特性（如 JSONB 操作符、窗口函数）
- 运维经验要求比 MySQL 更高（VACUUM、ANALYZE、连接池管理）
- 部分 ORM 对 PostgreSQL 高级特性的支持不够完善

### 实施计划

| 步骤 | 描述 | 负责人 | 预计时间 |
|------|------|--------|----------|
| 1 | 搭建 PostgreSQL 16 开发环境 | DevOps Engineer | 0.5 天 |
| 2 | 编写数据库连接和迁移工具配置 | Backend Engineer | 0.5 天 |
| 3 | 编写 PostgreSQL 开发规范文档 | Database Engineer | 1 天 |
| 4 | 团队培训（PostgreSQL 核心特性） | Database Engineer | 0.5 天 |

### 相关决策

- DEC-20260711-001: 选择微服务架构 — 每个微服务使用独立的数据库 Schema
- DEC-20260711-008: 数据库连接池配置规范 — 统一的连接池管理策略
```

### 示例 3：设计决策

```markdown
## DEC-20260711-005: 采用 API First 设计方法

### 元信息

| 字段 | 内容 |
|------|------|
| 决策编号 | DEC-20260711-005 |
| 决策日期 | 2026-07-11 |
| 决策者 | Tech Lead |
| 决策状态 | Accepted |
| 决策类型 | 设计决策 |
| 影响范围 | 前后端协作流程，API 设计流程 |

### 背景

在之前的项目中，API 文档在代码实现后才编写，导致前后端对接口定义理解不一致，集成测试阶段才发现问题，返工成本高。本次项目需要改进 API 设计流程，确保前后端对接口定义达成一致。

### 问题

API 设计应该遵循什么流程？是先写代码还是先写文档？

### 决策

采用 API First 设计方法，先编写 OpenAPI 3.0 规范文档，前后端共同评审并通过后，再开始代码实现。

### 理由

1. 前后端在开发前就接口定义达成一致，减少集成阶段的返工
2. OpenAPI 规范可以自动生成 Mock Server，前端可以并行开发
3. OpenAPI 规范可以自动生成 API 文档，文档始终与实现保持同步
4. 可以使用 Dredd/Schemathesis 等工具自动验证实现与规范的一致性
5. 符合团队 Documentation Driven Development 的原则

### 替代方案

| 方案 | 优点 | 缺点 | 放弃原因 |
|------|------|------|----------|
| Code First（先写代码） | 开发速度快，不需要额外编写规范文档 | 前后端理解不一致，集成阶段返工多 | 历史项目经验表明返工成本高 |
| API First（已选择） | 前后端一致，文档自动生成，减少返工 | 前期需要投入时间编写规范文档 | — |
| 口头约定 | 最快，无需文档 | 容易遗忘和误解，无法追溯 | 不适用于多团队协作 |

### 实施计划

| 步骤 | 描述 | 负责人 | 预计时间 |
|------|------|--------|----------|
| 1 | 编写 API 设计规范模板（OpenAPI 3.0 格式） | Backend Engineer | 0.5 天 |
| 2 | 编写第一个 API 规范文档（用户模块） | Backend Engineer | 1 天 |
| 3 | 前后端联合评审 API 规范 | 全团队 | 0.5 天 |
| 4 | 配置 OpenAPI 自动生成工具链（Mock Server + 文档生成） | Backend Engineer | 0.5 天 |

### 相关决策

- DEC-20260711-008: 选择 OpenAPI 3.0 作为 API 规范格式 — 具体的技术选型
```
